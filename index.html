Easy service discovery and routing for microservices

Install

Introduction

Baker Street simplifies scaling, testing, and upgrading microservices by:

  • automatically splitting traffic among all healthy services sharing the same name in the system
  • making load balancing more efficient and robust by using local load balancers
  • removing problematic instances from the rotation more quickly by using local health checkers
  • enabling canary testing for staged testing and deployment of service upgrades

Baker Street consists of three components:

  • Sherlock - an HAProxy-based routing system, with local instances corresponding to each instance of your application to determine where connections from that instance should go
  • Watson - a health checker with local instances corresponding to each instance of your application
  • Datawire Directory - a global service discovery mechanism that that receives availability information from each Watson instance and pushes changes in availability to local Sherlock instances as needed.

Alternatives

There are a number of different service discovery frameworks available. Why Baker Street?

  • Baker Street installs in just a few minutes. Try it!
  • None of these frameworks, with the exception of SmartStack, provide integrated routing. In other words, you have to update your application code to use the APIs provided by the service discovery framework. With Baker Street, you just need to point your application to the local proxy.
  • SmartStack (we're big fans) requires Zookeeper by default, and also exposes the HAProxy internal configuration. We've consciously decided to abstract this away (although you can always edit the configuration directly if you're so inclined.)
  • Most service discovery frameworks provide strong consistency semantics and elect to trade off with availability. Building highly available, strongly consistent systems is hard and complicated; we're not trying to solve this problem. This is because, in the service discovery realm, strong consistency isn't necessary -- when a microservice dies, you want to propagate the health information as quickly as possible, without waiting for consensus to propagate.
